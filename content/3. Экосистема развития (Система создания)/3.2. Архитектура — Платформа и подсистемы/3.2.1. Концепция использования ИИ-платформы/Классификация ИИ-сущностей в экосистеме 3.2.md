# Классификация ИИ-сущностей в экосистеме

## 0. Назначение документа

Документ определяет **четкую классификацию** всех ИИ-компонентов экосистемы и устанавливает границы между понятиями: ИИ-платформа, ИИ-системы, ИИ-ассистенты и ИИ-агенты.

**Адресаты:**
- Архитекторы и разработчики экосистемы
- Авторы документации
- Участники, создающие новые компоненты
- ИИ-агенты, работающие с репозиторием

---

## 1. Контекст и необходимость классификации

### 1.1. Зачем нужна классификация

В экосистеме развития используется множество ИИ-компонентов с разным назначением:
- Некоторые взаимодействуют с людьми (диалоговые ассистенты)
- Другие выполняют системные функции (обработка данных, координация)
- Третьи являются большими подсистемами (хранилища, аналитика)

**Без четкой классификации возникают проблемы:**
- Путаница в терминологии (что такое "агент" vs "ассистент"?)
- Неясная зона ответственности компонентов
- Сложность в именовании новых компонентов
- Противоречия в документации

### 1.2. Принципы классификации

Классификация строится по **трем критериям:**

1. **Уровень иерархии** — от системы в целом до отдельного исполнителя
2. **Адресат взаимодействия** — человек или другие системы
3. **Масштаб ответственности** — инфраструктура, подсистема, функция

```
┌─────────────────────────────────────────────────────────────┐
│  Уровень 1: ИИ-ПЛАТФОРМА                                    │
│  Вся экосистема как единая операционная система             │
└─────────────────────────────────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  Уровень 2: ИИ-СИСТЕМЫ                                      │

└─────────────────────────────────────────────────────────────┘
                          ▼
       ┌─────────────────────────────────────┐
       │  Уровень 3а: ИИ-АССИСТЕНТЫ          │
       │  Человеко-ориентированные,          │
       │  диалоговые, персонализированные    │
       └─────────────────────────────────────┘
                          ▼
       ┌─────────────────────────────────────┐
       │  Уровень 3б: ИИ-АГЕНТЫ              │
       │  Системно-ориентированные,          │
       │  исполняют конкретные функции       │
       └─────────────────────────────────────┘
```

## 3. Уровень 1: ИИ-Платформа

### 3.1. Определение

**ИИ-Платформа** — это вся совокупность ИИ-компонентов экосистемы, работающая как единая **операционная система** для развития человека.

### 3.2. Характеристики

- **Единственная** в экосистеме (не может быть двух ИИ-платформ)
- Включает все компоненты уровней 2, 3а, 3б
- Является как **инструментом для экосистемы**, так и **ее частью**
- Обеспечивает связность всех компонентов через единые протоколы (MCP, Event Bus)

### 3.3. Назначение

- Координация всех ИИ-компонентов
- Обеспечение единой модели данных (Цифровой двойник)
- Управление жизненным циклом ИИ-сущностей
- Предоставление общей инфраструктуры (идентификация, безопасность, аудит)

### 3.4. Примеры компонентов платформенного уровня

- **Операционная система ИИ-платформы** — ядро координации
- **Централизованное хранилище** — единое состояние
- **MCP-инфраструктура** — протокол взаимодействия
- **Event Bus** — шина событий

---

## 4. Уровень 2: ИИ-Системы

### 4.1. Определение

**ИИ-Система** — это крупная автономная подсистема экосистемы, которая:
- Имеет собственную инфраструктуру (БД, API, процессы)
- Решает обособленную задачу предметной области
- Может включать несколько агентов и компонентов
- Взаимодействует с другими системами через стандартные интерфейсы

### 4.2. Критерии отнесения к ИИ-системам

Компонент является **ИИ-системой**, если:
1. Имеет собственное хранилище данных (БД, граф, индекс)
2. Предоставляет API для внешних систем
3. Включает несколько взаимосвязанных агентов или подсистем
4. Может работать относительно автономно

### 4.3. Список ИИ-систем экосистемы

| ИИ-Система | Назначение | Ключевые компоненты |
|------------|------------|---------------------|
| **Цифровой двойник** | Хранение состояния, траектории, артефактов человека | SurrealDB, MCP-серверы, схема данных |
| **Хаб активностей** | Сбор и агрегация событий от всех подсистем | Event Collector, Aggregator, Analytics |
| **Распределитель токенов** | Начисление и распределение токенов за активности | Rules Engine, Wallet Manager, Accounting |
| **Эпистемический граф** | Структура знаний, понятий, связей руководств | Knowledge Graph, Ontology, Reasoner |
| **Система аналитики** | Обработка метрик, построение инсайтов, отчеты | Data Warehouse, ML Models, Dashboards |
| **Система персонализации** | Адаптация контента под человека | Profile Engine, Content Adapter, Preferences |

### 4.4. Принципы работы ИИ-систем

- Каждая система имеет **явный контракт** (API, события, данные)
- Системы **слабо связаны** (loose coupling) через Event Bus и MCP
- Системы **независимо масштабируются**
- Системы могут включать агенты для внутренней логики

## 5. Уровень 3а: ИИ-Ассистенты

### 5.1. Определение

**ИИ-Ассистент** — это диалоговый ИИ-компонент, который:
- Взаимодействует **напрямую с человеком** через чат/интерфейс
- Персонализирован под конкретную **роль человека**
- Использует данные Цифрового двойника и методику
- Помогает человеку в **развитии, обучении, рефлексии**

### 5.2. Критерии отнесения к ИИ-ассистентам

Компонент является **ИИ-ассистентом**, если:
1. Основной интерфейс — **диалог с человеком**
2. Персонализирован под **роль** (Ученик, Интеллектуал, Профессионал)
3. Работает с **траекторией развития** конкретного человека
4. Не выполняет системных функций (не обрабатывает данные пакетно)

### 5.3. Список ИИ-ассистентов экосистемы

| ИИ-Ассистент | Роль | Назначение | Статус |
|--------------|------|------------|--------|
| **Проводник (Ассистент Ученика)** | Ученик | Персональный маршрут развития, ступени Ученика, регулярность | ✅ В разработке |
| **Ассистент Интеллектуала** | Интеллектуал | Компиляция знаний, построение мировоззрения | 🔮 Планируется |
| **Ассистент Профессионала** | Профессионал | Применение знаний в практике, создание систем | 🔮 Планируется |
| **Ассистент Исследователя** | Исследователь | Научный метод, эксперименты, публикации | 🔮 Планируется |
| **Ассистент Просветителя** | Просветитель | Передача знаний, наставничество, обучение других | 🔮 Планируется |

### 5.4. Архитектура ИИ-ассистента (на примере Проводника)

ИИ-ассистент работает по модели **"оркестратор + агенты"**:

```
┌──────────────────────────────────────────────────────────┐
│  Проводник (Ассистент Ученика)                           │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Диалоговый интерфейс (ChatGPT, Telegram, LMS)    │  │
│  └────────────────────────────────────────────────────┘  │
│                         ▼                                 │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Оркестратор (роль R.GrowthGuide)                 │  │
│  │  - Понимает контекст диалога                      │  │
│  │  - Определяет нужные действия                     │  │
│  │  - Вызывает агенты через MCP                      │  │
│  └────────────────────────────────────────────────────┘  │
│                         ▼                                 │
│  Использует агенты:                                      │
│  - AGT.RouteGuide (формирование маршрута)                │
│  - AGT.TrajectoryPlanner (планирование траектории)       │
│  - AGT.DailyAssignmentBuilder (задания дня)              │
│  - AGT.ContentPersonalizer (персонализация)              │
│  - AGT.Evaluator (оценка артефактов)                     │
│  - AGT.RhythmKeeper (поддержка ритма)                    │
└──────────────────────────────────────────────────────────┘
```

**Ключевое отличие:** ассистент — это "лицо" для человека, а агенты — "руки и ноги" ассистента.

## 6. Уровень 3б: ИИ-Агенты

### 6.1. Определение

**ИИ-Агент** — это системный ИИ-компонент, который:
- Выполняет **конкретную функцию** или задачу
- Взаимодействует с **данными и другими системами** (не с человеком напрямую)
- Вызывается **другими агентами, ассистентами или системами**
- Имеет **четкий контракт**: входы, выходы, инструменты (TOOL.*)

### 6.2. Критерии отнесения к ИИ-агентам

Компонент является **ИИ-агентом**, если:
1. Основная задача — **обработка данных** или **выполнение функции**
2. Не имеет прямого диалогового интерфейса с человеком
3. Вызывается **программно** (через API, MCP, Event)
4. Имеет **явный контракт** входов и выходов

### 6.3. Группы ИИ-агентов

Согласно документу [[Список ИИ-агентов 3.2]], все агенты организованы в **5 групп**:

#### Группа 1: Ядро выполнения и координации
- `AGT.SessionOrchestrator` — оркестрация сессий
- `AGT.PolicyGatekeeper` — права и политики доступа
- `AGT.ToolRouter` — маршрутизация инструментов (MCP Hub)

#### Группа 2: Сбор фактов (ingestion → Twin)
- `AGT.ActivityIngestor` — сбор событий из систем
- `AGT.ArtifactExtractor` — извлечение артефактов из текстов
- `AGT.DataQualitySteward` — контроль качества данных

#### Группа 3: Обучение и развитие (Twin → план → действие → Twin)
- `R.GrowthGuide` + `AGT.RouteGuide` — проводник по маршруту
- `AGT.TrajectoryPlanner` — планирование траектории
- `AGT.DailyAssignmentBuilder` — генерация заданий дня
- `AGT.ContentPersonalizer` — персонализация контента
- `R.Evaluator` + `AGT.Evaluator` — оценка артефактов
- `R.Methodist` + `AGT.Methodist` — улучшение методики

#### Группа 4: Поддержка ритма и взаимодействия
- `AGT.RhythmKeeper` — ритм и напоминания
- `AGT.SupportTriage` — триаж запросов поддержки

#### Группа 5: Аналитика, наблюдаемость, безопасность
- `AGT.ProgressAnalyst` — анализ прогресса
- `AGT.AnomalyRiskMonitor` — мониторинг аномалий
- `AGT.AuditTrace` — аудит операций

### 6.4. Нотация агентов

В документации используется единая нотация:
- `AGT.*` — исполняемый агент (executable agent)
- `R.*` — роль (role), которую может исполнять агент или человек
- `TOOL.*` — инструмент, доступный через MCP

**Примеры:**
- `AGT.RouteGuide` — агент формирования маршрута
- `R.GrowthGuide` — роль проводника (может быть как агентом, так и человеком-ментором)
- `TOOL.ReadTwin` — инструмент чтения Цифрового двойника

## 7. Сравнительная таблица уровней

| Критерий | ИИ-Платформа | ИИ-Система | ИИ-Ассистент | ИИ-Агент |
|----------|--------------|------------|--------------|----------|
| **Количество** | 1 (единственная) | ~5-10 (ограниченное) | ~5 (по ролям) | ~15-20 (расширяемое) |
| **Адресат** | Вся экосистема | Системы и агенты | Человек | Данные, системы |
| **Интерфейс** | Инфраструктура | API, Events | Диалог (чат) | API, MCP, Events |
| **Автономность** | Полная | Высокая | Средняя | Низкая (вызывается) |
| **Персонализация** | Нет | Нет | Высокая (под человека) | Нет |
| **Хранилище** | Общее (Twin) | Собственное | Использует Twin | Использует Twin |
| **Масштаб** | Вся экосистема | Предметная область | Траектория человека | Одна функция |

## 8. Примеры применения классификации

### 8.1. Сценарий: Человек общается с Проводником

```
1. Человек → [ИИ-Ассистент: Проводник]
   "У меня ощущение, что топчусь на месте. Что делать?"

2. [Проводник] → [AGT.RouteGuide]
   "Получи данные о человеке и сформируй рекомендации"

3. [AGT.RouteGuide] → [TOOL.ReadTwin] (MCP)
   "Читай данные Twin: слоты, задания, активность"

4. [AGT.RouteGuide] → [TOOL.ReadMethodology] (MCP)
   "Читай методику: ступени Ученика, практики"

5. [AGT.RouteGuide] → [AGT.TrajectoryPlanner]
   "Построй план на основе данных и методики"

6. [AGT.TrajectoryPlanner] → [TOOL.WriteTwin] (MCP)
   "Запиши новый шаг маршрута в Twin"

7. [Проводник] → Человек
   "Вот твой план на неделю: ..."
```

**Классификация компонентов:**
- **Проводник** — ИИ-ассистент (диалог с человеком)
- **AGT.RouteGuide, AGT.TrajectoryPlanner** — ИИ-агенты (обработка данных)
- **TOOL.ReadTwin, TOOL.WriteTwin** — инструменты (доступ к ИИ-системе "Цифровой двойник")
- **Цифровой двойник** — ИИ-система (хранилище + API)

### 8.2. Сценарий: Начисление токенов за активность

```
1. [LMS] → [Event Bus] (ИИ-Платформа)
   Событие: "Пользователь X завершил модуль Y"

2. [Хаб активностей] (ИИ-Система) → [AGT.ActivityIngestor]
   "Обработай событие и запиши в Twin"

3. [AGT.ActivityIngestor] → [TOOL.WriteTwin]
   "Запиши событие E.* в Twin пользователя X"

4. [Распределитель токенов] (ИИ-Система) → [AGT.TokenCalculator]
   "Вычисли токены за событие E.*"

5. [AGT.TokenCalculator] → [TOOL.WriteWallet]
   "Начисли N токенов на кошелек пользователя X"
```

**Классификация компонентов:**
- **Event Bus** — компонент ИИ-платформы
- **Хаб активностей, Распределитель токенов** — ИИ-системы
- **AGT.ActivityIngestor, AGT.TokenCalculator** — ИИ-агенты

## 9. Правила именования компонентов

### 9.1. ИИ-Платформа
- Название: "ИИ-Платформа" (единственное, без вариантов)
- В документах: всегда с заглавной буквы
- Альтернативы: "Платформа", "ИИ-инфраструктура экосистемы"

### 9.2. ИИ-Системы
- Формат: `Название системы` (без префикса "ИИ-")
- Примеры: "Цифровой двойник", "Хаб активностей", "Распределитель токенов"
- В коде/API: snake_case (`digital_twin`, `activity_hub`)

### 9.3. ИИ-Ассистенты
- Формат: `Название ассистента` или `Ассистент [Роли]`
- Примеры: "Проводник", "Ассистент Ученика", "Ассистент Интеллектуала"
- В коде: `assistant_learner`, `assistant_intellectual`

### 9.4. ИИ-Агенты
- Формат: `AGT.NameInPascalCase` или `R.RoleInPascalCase`
- Примеры: `AGT.RouteGuide`, `R.GrowthGuide`, `AGT.DataQualitySteward`
- В документах: всегда с префиксом, чтобы отличать от других сущностей

### 9.5. Инструменты (Tools)
- Формат: `TOOL.ActionTarget`
- Примеры: `TOOL.ReadTwin`, `TOOL.WriteWallet`, `TOOL.FetchMethodology`
