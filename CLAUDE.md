## 1. Тип репозитория

**Downstream-governance** — управленческая сборка экосистемы, оформленная по S2R.

**НЕ является source-of-truth** — знания берутся из Pack'ов.

### 1.1. Что содержит этот репозиторий

| Категория | Примеры |
|-----------|---------|
| **Справочный контент** | Что есть, где искать, ссылки на Pack'и |
| **Реестры** | Объекты внимания, создаваемые системы, агенты, партнёры |
| **Планы и стратегии** | Q1 2026 roadmap, приоритеты, milestone'ы |
| **Процессы взаимодействия** | Кто с кем, как, SLA, ритуалы |
| **Организационная структура** | Роли команды, ответственности |
| **Управленческие решения** | ADR, трассировка решений |

### 1.2. Что НЕ содержит (→ Pack)

| Контент | Куда переносить |
|---------|-----------------|
| Знания предметной области | → spf-*-pack |
| Архитектуры как знание | → spf-*-pack |
| Методы (способы действия) | → spf-*-pack |
| Характеристики, состояния | → PACK-personal |
| Work products, failure modes | → spf-*-pack |

### 1.3. Pack-репозитории (source-of-truth)

| Pack | Область | Что содержит |
|------|---------|--------------|
| [PACK-personal](../PACK-personal/) | Созидатель | Характеристики, состояния, роли, программы развития |
| [PACK-ecosystem](../PACK-ecosystem/) | Экосистема | Знания об экосистеме, компоненты, бесшовность |
| [PACK-digital-platform](../PACK-digital-platform/) | ИТ-платформа | Архитектура платформы, цифровой двойник, MCP, агенты |

## 4. Правила работы

### 4.1. Ops-First Rule

**При изменениях в репозитории ВСЕГДА загружай документы из 0.OPS/ ПЕРВЫМИ.**

Priority Queue:
1. `01-kernels-model.md` — структура ядер
2. `02-document-families.md` — семейства F1-F9
3. `07-naming.md` — правила именования
4. `05-glossary.md` — терминология

### 4.2. Правило нумерации X1/X2/X3

```
X1 = Надсистема (физическая вложенность)
X2 = SoI (система интереса)
X3 = Система создания
```

**НЕЛЬЗЯ** менять нумерацию в зависимости от контекста.

### 4.3. Система ≠ Эпистема

**Ядро** — только для систем (физических сущностей).

**ЗАПРЕЩЕНО:** Ядра по эпистемам (`E.Экономика/`, `G.Продвижение/`).

### 4.4. Иерархия источников знаний (Fallback Chain)

Этот репозиторий — **Downstream**. Источники знаний по приоритету:

```
1. Pack'и (source-of-truth)
   └── PACK-personal, PACK-ecosystem, PACK-digital-platform
2. SPF (форма и процесс)
   └── ~/Github/SPF/
3. FPF (первые принципы) — только если нет в SPF
   └── ~/Github/FPF/FPF-Spec.md
```

**Правило:** Если чего-то нет на текущем уровне — смотри выше.

**Когда читать FPF напрямую:**
- Только если вопрос не покрыт в SPF
- Для понимания базовых различений (A.7, A.1.1)

**Документация интеграции S2R↔FPF:** `0.OPS/0.4.FPF-Integration/`

### 4.5. Правило чистоты Governance

> **КРИТИЧЕСКОЕ ПРАВИЛО:** Этот репозиторий содержит ТОЛЬКО governance-контент.

**При создании/изменении документа спроси:**
1. "Это знание домена или управление проектом?"
2. Если знание → отправь в соответствующий Pack
3. Если управление → создай здесь со ссылкой на Pack

**Тест:** Можно ли использовать этот контент в другом проекте/экосистеме?
- Да → Это Pack-контент, нужна миграция
- Нет (привязано к этой команде/срокам) → Governance, можно оставить

### 4.6. Формат Architecture-папок (X2.2)

Architecture-папки (A2.2, B2.2, C2.2) содержат полную структуру s2r:

| Элемент | Содержит |
|---------|----------|
| `_index.md` | Навигация, ссылки на Pack, реестр |
| Файлы в папке | Governance-описания, связи, контекст |
| Подпапки | Детализация (со своими _index.md и контентом) |

**Шаблон _index.md (навигационный документ):**

```markdown
# X.X.N. Название раздела

> **Source-of-truth**: [spf-*-pack](link)
>
> Формализованное знание находится в Pack-репозитории.
> Этот документ содержит ссылки, реестр и навигацию.

## Назначение
[Краткое описание раздела]

## Реестр объектов внимания

| Объект | Описание | Source |
|--------|----------|--------|
| ... | ... | [spf-*-pack/.../](link) |

## Governance-контент (локальный)
[Реестры, инструменты, процессы — то, что привязано к проекту]

## Подразделы
[Навигация по подпапкам]
```

### 4.7. Миграция Pack-контента

**Если нашёл Pack-контент в этом репозитории:**

1. Идентифицируй тип контента (знание/метод/характеристика/...)
2. Определи целевой Pack (personal/ecosystem/digital-platform)
3. Предложи пользователю план миграции:
   - Что переносится
   - Куда переносится
   - Что остаётся здесь (ссылка + реестр)

**Индикаторы Pack-контента:**
- Описание "как устроено" (а не "что делаем")
- Знания, архитектуры, характеристики
- Методы (способы действия)
- Универсальные концепции (не привязаны к срокам)

## 8. Миграция контента в Pack-репозитории

### 8.1. Принцип работы с Pack-контентом

**Правило:** Pack-контент (знания, методы, характеристики) хранится в Pack-репозиториях (source-of-truth).
В DS-ops остаётся governance-контент со ссылками на Pack.

| Ядро | Source-of-truth | Что в Pack |
|------|-----------------|------------|
| A (Созидатель) | PACK-personal | Знания, характеристики, состояния |
| B (Экосистема) | PACK-ecosystem | Знания об экосистеме, компоненты |
| C (ИТ-платформа) | PACK-digital-platform | Архитектура платформы, агенты |

**Структура сохраняется по s2r.** Каждое ядро содержит полную иерархию X1/X2/X3 × Meaning/Architecture/Operations.

### 8.2. Структура репозитория

**Структура полностью соответствует s2r** — со всеми папками и подпапками.

Каждое ядро содержит полную иерархию:
```
A.Systems-Builder/
├── A1.Teams-and-Communities/
│   ├── A1.1.Meaning/       # Надсистема × Meaning
│   ├── A1.2.Architecture/  # Надсистема × Architecture
│   └── A1.3.Operations/    # Надсистема × Operations
├── A2.Systems-Builder/
│   ├── A2.1.Meaning/       # SoI × Meaning
│   ├── A2.2.Architecture/  # SoI × Architecture
│   │   ├── 2.2.1. Программы развития/
│   │   │   ├── _index.md           # Навигация + ссылки на Pack
│   │   │   ├── О программах...     # Governance-описание
│   │   │   └── 2.2.1.1. .../       # Подразделы с контентом
│   │   ├── 2.2.2. Роли созидателя/

    ├── A3.2.Architecture/  # Constructor × Architecture
    └── A3.3.Operations/    # Constructor × Operations
```

### 8.3. Роль _index.md файлов

Файлы `_index.md` в подпапках — это **навигационные документы**, которые:
1. Содержат ссылки на source-of-truth в Pack
2. Описывают назначение раздела
3. Дают реестр объектов внимания
4. Указывают на подразделы и governance-контент

**Важно:** _index.md НЕ заменяет весь контент папки. В подпапках находится:
- Governance-контент (реестры, описания, связи)
- Контекстные документы, связывающие области знаний
- Рабочие материалы и описания общих идей

### 8.4. Правило для контента

| Тип контента | Где находится | Примеры |
|--------------|---------------|---------|
| Формализованное знание | Pack (source-of-truth) | Знания, методы, характеристики |
| Ссылки и навигация | `_index.md` | Реестр, ссылки на Pack |
| Governance-описания | Файлы в подпапках | Описания, связи, контекст |
| Процессы и планы | Operations папки | SLA, ритуалы, планы |
