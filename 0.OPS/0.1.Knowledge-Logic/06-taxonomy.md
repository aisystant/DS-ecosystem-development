# Таксономия документов (Document Taxonomy)

## Назначение документа

Этот документ определяет **систему классификации документов** в S2R репозитории. Таксономия обеспечивает единообразную категоризацию всех артефактов знаний.

---

## Типы документов

### По содержанию

| Тип | Код | Описание | Примеры |
|-----|-----|----------|---------|
| **Концептуальные** | `concept` | Визия, миссия, ценности | Манифест, стратегия, vision statement |
| **Спецификации** | `spec` | Требования, архитектура | ADR, system spec, API spec |
| **Процессные** | `process` | Регламенты, инструкции | Онбординг, deployment guide, SOP |
| **Методологические** | `method` | Методы, практики | DDD, Event Storming, JTBD |
| **Справочные** | `reference` | Глоссарии, каталоги | Glossary, API reference, cheat sheet |
| **Отчётные** | `report` | Отчёты, метрики | ИИ-отчёты, дашборды, аналитика |
| **Черновики** | `draft` | Незавершённые идеи | Заметки в 0.9.Inbox/ |

### По статусу зрелости

| Статус | Описание | Действия |
|--------|----------|----------|
| **stub** | Заглушка, структура без содержания | Заполнить контентом |
| **draft** | Черновик, в процессе разработки | Ревью и доработка |
| **review** | На ревью | Проверка и согласование |
| **stable** | Стабильный, используется | Поддержка актуальности |
| **deprecated** | Устарел, не используется | Архивировать или обновить |

### По уровню формальности (FPF Trust Calculus)

| Уровень F | Описание | Применение |
|-----------|----------|------------|
| **0-2** | Скетч, идея | Ранние гипотезы, мозговой штурм |
| **3-5** | Рабочий документ | Текущая работа команды |
| **6-7** | Согласованный | Утверждённые спецификации |
| **8-9** | Формальное доказательство | Критичные архитектурные решения |

## Оси классификации

### Ось 1: Семейство документов (F0-F9)

**Описание**: Классификация по матрице "Системы × Роли"

**Значения**: F0 (мета), F1-F9 (матрица 3×3)

**Пример**:
```yaml
family: F5  # Architecture × SoI
```

**Правило**: Семейство определяет целевую аудиторию и разработчика (см. [[02-document-families.md]]).

### Ось 2: Ядро (Kernel)

**Описание**: К какому ядру относится документ

**Значения**: F0 (0.OPS), A, B, C, ... (ядра)

**Пример**:
```yaml
kernel: C  # Ядро C.ИТ-платформа-Экосистемы
```

### Ось 3: Система (System)

**Описание**: К какой системе в ядре относится документ

**Значения**: X1 (надсистема), X2 (SoI), X3 (создания)

**Пример**:
```yaml
system: C2  # SoI ядра C
```

### Ось 4: Роль (Role)

**Описание**: С какой роли описывается система

**Значения**: Meaning, Architecture, Operations

**Пример**:
```yaml
role: Architecture  # Архитектурный взгляд
```

### Ось 5: Целевая аудитория (Target Audience)

**Описание**: Для кого предназначен документ (КРИТИЧЕСКИЙ атрибут)

**Значения**: Зависят от семейства (см. таблицу ниже)

**Пример**:
```yaml
target_audience:
  - "Методологи"
  - "Наставники"
```

### Ось 6: Разработчик (Developer)

**Описание**: Кто создаёт и поддерживает документ

**Значения**: Роли команды (см. таблицу ниже)

**Пример**:
```yaml
developer_role: "Методолог"
```

---

## Матрица "Семейство → Аудитория → Разработчик"

| Семейство | Целевые читатели | Разработчик | НЕ для |
|-----------|------------------|-------------|--------|
| **F0** | ИИ-агенты, редакторы | Администратор хранилища | Ученики, инвесторы |
| **F1** | Потенциальные участники | Маркетолог, Продвиженец | Инвесторы (→F7) |
| **F2** | Пользователи продукта | Владелец продукта, UX-дизайнер | Инвесторы (→F7) |
| **F3** | Партнёры, регуляторы, медиа | PR-менеджер, Коммуникатор | Ученики (→F4/F6) |
| **F4** | Ученики, созидатели | **Визионер** | Инвесторы (→F7), разработчики (→F8) |
| **F5** | Наставники, методологи | Методолог, Проектировщик обучения | Ученики напрямую (→F4/F6) |
| **F6** | Ученики, наставники | Деканат, Служба сопровождения | Инвесторы (→F7) |
| **F7** | Инвесторы, стратеги | **Бизнесмен**, Экономист | Ученики (→F4) |
| **F8** | DevOps, техлиды | Архитектор, Разработчик | Ученики (→F4), инвесторы (→F7) |
| **F9** | Операторы, финансисты | HR, Администратор | Ученики (→F4) |

## Эпистемические уровни (FPF)

### Trust Calculus (B.3)

**Формальность (F)**: 0-9
- 0 = скетч, идея
- 5 = рабочий документ
- 9 = формальное доказательство

**Область применения (G)**: universal | domain | local-edge
- universal = универсальные принципы
- domain = доменная специфика
- local-edge = локальные особенности

**Надёжность (R)**: 0.0-1.0
- Уверенность в утверждениях документа

**Пример**:
```yaml
trust:
  F: 5  # Рабочий документ
  G: "domain"  # Доменная специфика
  R: 0.8  # Высокая уверенность
```

### Epistemic Stage (B.5.1)

| Стадия | Описание | Действия |
|--------|----------|----------|
| **explore** | Исследование | Генерация гипотез |
| **shape** | Формирование | Структурирование знаний |
| **evidence** | Сбор доказательств | Валидация гипотез |
| **operate** | Эксплуатация | Использование в работе |

**Пример**:
```yaml
epistemic_stage: shape  # Формируем структуру
```

## Правила классификации

### Правило 1: Обязательные атрибуты

Каждый документ ДОЛЖЕН иметь:
```yaml
---
family: F5
kernel: C
system: C2
role: Architecture
target_audience:
  - "Методологи"
status: stable
---
```

### Правило 2: Соответствие аудитории семейству

**Проверка**: target_audience ДОЛЖЕН совпадать с семейством.

**❌ Ошибка**:
```yaml
family: F1  # Для потенциальных участников
target_audience:
  - "Инвесторы"  # ← ОШИБКА! Инвесторы → F7
```

**✅ Правильно**:
```yaml
family: F1
target_audience:
  - "Потенциальные участники"
  - "Широкая аудитория"
```

### Правило 3: Один документ = одно семейство

Документ НЕ может принадлежать двум семействам одновременно.

**Решение**: Если документ актуален для двух аудиторий → создать два документа.

### Правило 4: Связи документов

Документы связываются через:
- **depends_on** — зависит от (родительские документы)
- **affects** — влияет на (дочерние документы)
- **related** — связан с (горизонтальные связи)
- **tests** — тестирует (для тестовых документов)

**Пример**:
```yaml
depends_on:
  - "01-kernels-model.md"
  - "03-our-systems-map.md"
affects:
  - "A.Созидатель/A2.Созидатель/A2.2.Architecture/model.md"
related:
  - "07-naming.md"
```

## Типичные ошибки классификации

### Ошибка 1: Смешение аудиторий

**Проблема**: Документ в F4 описывает техническую архитектуру.

**Исправление**: Техническая архитектура → F8 (для разработчиков).

### Ошибка 2: Неправильное семейство

**Проблема**: Документ об инвестициях в F1.

**Исправление**: Инвестиции → F7 (для инвесторов).

### Ошибка 3: Отсутствие target_audience

**Проблема**: Frontmatter без target_audience.

**Исправление**: Всегда указывать target_audience (критический атрибут).

## Шаблон Frontmatter

```yaml
---
# === БАЗОВЫЕ МЕТАДАННЫЕ ===
type: doc | spec | process | method | reference | report
family: F5                    # F0-F9 по матрице ролей
kernel: C                     # A/B/C или F0
system: C2                    # X1/X2/X3
role: Architecture            # Meaning/Architecture/Operations
status: draft | review | stable | deprecated
created: 2025-01-14
modified: 2025-01-14

# === ЦЕЛЕВЫЕ РОЛИ (ОБЯЗАТЕЛЬНО) ===
target_audience:              # Для кого предназначен
  - "Методологи"
  - "Наставники"
developer_role: "Методолог"   # Кто создаёт

# === FPF: TRUST CALCULUS ===
trust:
  F: 5                        # Formality: 0-9
  G: "domain"                 # ClaimScope: universal | domain | local-edge
  R: 0.8                      # Reliability: 0.0-1.0

# === FPF: EPISTEMIC STAGE ===
epistemic_stage: shape        # explore | shape | evidence | operate

# === СВЯЗИ ДОКУМЕНТОВ ===
depends_on:                   # От каких документов зависит
  - "path/to/parent.md"
affects:                      # На какие документы влияет
  - "path/to/child.md"
related:                      # Связанные документы
  - "path/to/related.md"

# === ТЕГИ ===
tags:
  - kernel-c
  - architecture
  - ai-platform
---
```

## Автоматическая валидация

ИИ-агенты проверяют:
1. ✅ Наличие всех обязательных атрибутов
2. ✅ Соответствие target_audience семейству
3. ✅ Корректность ссылок (depends_on, affects)
4. ✅ Уникальность документа (нет дубликатов)
5. ✅ Правильное расположение в структуре
