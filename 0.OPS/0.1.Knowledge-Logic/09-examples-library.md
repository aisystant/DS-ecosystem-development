# Роль: Созидатель-Ученик

## Контекст

**Созидатель-Ученик** — это человек, активно развивающий свои компетенции для достижения мастерства в выбранной области.

## Зачем эта роль?

Роль созидателя-ученика важна, потому что:
- Определяет **начальную точку пути** развития
- Обеспечивает **фокус на обучении** и практике
- Создаёт **мотивацию** для продвижения

## Ключевые характеристики

1. **Открытость к обучению**: готовность экспериментировать и ошибаться
2. **Фокус на практике**: применение знаний в реальных проектах
3. **Рефлексия**: осознание своего прогресса

## См. также

- [[C.Модель-Созидателя.md]] — полная модель созидателя
- [[R.Созидатель-Выпускник.md]] — следующий этап развития
- [[../A2.2.Architecture/C.Путь-Развития.md]] — структура пути

# Компонент: API платформы

## Обзор

REST API платформы обеспечивает взаимодействие между клиентами (web, mobile) и бэкендом.

## Архитектура

### Эндпоинты

**Аутентификация**:
- `POST /auth/login` — вход
- `POST /auth/refresh` — обновление токена

**Ученики**:
- `GET /students` — список учеников
- `GET /students/:id` — данные ученика
- `PUT /students/:id` — обновление профиля

**Компетенции**:
- `GET /competencies` — каталог компетенций
- `POST /assessments` — оценка компетенции

### Технологии

- **Фреймворк**: FastAPI (Python 3.11)
- **Аутентификация**: JWT токены
- **Валидация**: Pydantic schemas
- **Документация**: OpenAPI / Swagger

## Пример 3: Связи документов

### Дерево зависимостей

Правильная структура зависимостей образует **DAG (Directed Acyclic Graph)**:

```
01-kernels-model.md (корень)
├── depends_on: []
└── affects:
    ├── 02-document-families.md
    └── 03-our-systems-map.md

02-document-families.md
├── depends_on: [01-kernels-model.md]
└── affects:
    ├── 06-taxonomy.md
    └── A2.2.Architecture/C.Модель-Созидателя.md

06-taxonomy.md
├── depends_on: [02-document-families.md, 04-ontology.md]
└── affects:
    └── A2.2.Architecture/C.Модель-Созидателя.md

A2.2.Architecture/C.Модель-Созидателя.md
├── depends_on: [02-document-families.md, 06-taxonomy.md]
└── affects:
    ├── A2.1.Meaning/R.Созидатель-Ученик.md
    └── A2.3.Operations/E.Обучение.md
```

**Интерпретация**:
- **Корневые документы** (01, 04) не зависят ни от кого
- **Промежуточные документы** (02, 06) зависят от корневых и влияют на доменные
- **Листовые документы** (R., E., C.) зависят от промежуточных

### Горизонтальные связи

Документы одного уровня связаны через `related`:

```yaml
# A2.1.Meaning/R.Созидатель-Ученик.md
related:
  - "R.Созидатель-Выпускник.md"    # Следующая стадия
  - "../A2.2.Architecture/C.Модель-Созидателя.md"  # Архитектура

# A2.1.Meaning/R.Созидатель-Выпускник.md
related:
  - "R.Созидатель-Ученик.md"       # Предыдущая стадия
  - "R.Созидатель-Мастер.md"       # Следующая стадия
```

## Пример 6: Миграция между ядрами

### Сценарий: Переход одноядерной структуры в многоядерную

**Было (до миграции)**:
```
AISYSTANT/
├── A1.Мир/
├── A2.Созидатель/
└── A3.Экосистема/
```

**Стало (после миграции)**:
```
A.Созидатель/
├── A1.Команды/
├── A2.Созидатель/
└── A3.Экосистема-Развития/

B.Экосистема/
├── B1.Мир/
├── B2.Экосистема/
└── B3.Система-Создания/

C.ИТ-Платформа-Экосистемы/
├── C1.Экосистема/
├── C2.ИТ-Платформа/
└── C3.ИТ-Служба/
```

### Процесс миграции

1. **Анализ**: Определить SoI каждого ядра (Созидатель, Экосистема, Платформа).
2. **Создание структуры**: Создать папки A/, B/, C/ с полной 3×3 матрицей.
3. **Перемещение файлов**:
   - `A2.Созидатель/` → `A.Созидатель/A2.Созидатель/` (SoI ядра A)
   - Файлы об экосистеме → `B.Экосистема/B2.Экосистема/` (SoI ядра B)
   - Файлы о платформе → `C.ИТ-Платформа-Экосистемы/C2.ИТ-Платформа/` (SoI ядра C)
4. **Обновление Frontmatter**:
   ```yaml
   # Было:
   kernel: AISYSTANT
   system: A2
   
   # Стало:
   kernel: A
   system: A2
   ```
5. **Обновление ссылок**: Все wikilinks и depends_on обновить на новые пути.
6. **Архивирование**: `old/AISYSTANT/` → сохранить для истории.

**Отчёт о миграции**: `artifacts/migration-report-2025-01-15.md`

## Пример 7: Работа с FPF (Formality-Practice-Fit)

### Trust Calculus

**Низкая формальность (F=2)** — ранняя идея:
```yaml
trust:
  F: 2          # Скетч
  G: "local"    # Локальная гипотеза
  R: 0.5        # Средняя уверенность
status: draft
```

**Средняя формальность (F=5)** — рабочий документ:
```yaml
trust:
  F: 5          # Рабочий документ
  G: "domain"   # Доменная специфика
  R: 0.8        # Высокая уверенность
status: stable
```

**Высокая формальность (F=8)** — критическая архитектура:
```yaml
trust:
  F: 8          # Формальная спецификация
  G: "universal" # Универсальные принципы
  R: 0.95       # Очень высокая уверенность
status: stable
```

### Epistemic Stage

**Explore** — исследование:
```yaml
epistemic_stage: explore
status: draft
# Генерируем гипотезы, пробуем разные подходы
```

**Shape** — формирование:
```yaml
epistemic_stage: shape
status: review
# Структурируем знания, выбираем лучший подход
```

**Evidence** — сбор доказательств:
```yaml
epistemic_stage: evidence
status: stable
# Валидируем гипотезы, собираем данные
```

**Operate** — эксплуатация:
```yaml
epistemic_stage: operate
status: stable
# Используем в повседневной работе
```

## Использование библиотеки

### Для разработчиков

**Перед созданием нового ядра**:
1. Изучить Пример 1 (Созидатель) и Пример 2 (ИТ-Платформа)
2. Определить SoI нового ядра
3. Определить отношения (∋, ⊃, →) по аналогии
4. Создать структуру 3×3 матрицы

**Перед созданием документа**:
1. Определить семейство (F0-F9) по Примеру 4
2. Выбрать шаблон (Пример 9)
3. Проверить именование (Пример 5)
4. Определить зависимости (Пример 3)

### Для ревьюеров

**При code review**:
1. Сравнить структуру с Примерами 1-2
2. Проверить именование по Примеру 5
3. Проверить семейство по Примеру 4
4. Проверить зависимости по Примеру 3
5. Убедиться, что нет антипаттернов (см. [[08-anti-patterns.md]])

### Для ИИ-агентов

**При генерации нового контента**:
1. Загрузить [[01-kernels-model.md]], [[04-ontology.md]], [[09-examples-library.md]]
2. Определить kernel, system, role для нового документа
3. Использовать шаблон из Примера 9
4. Проверить соответствие Примерам 1-5
5. Валидировать против [[08-anti-patterns.md]]

## Расширение библиотеки

### Добавление новых примеров

**Процесс**:
1. Создать черновик примера в формате выше
2. Добавить в соответствующий раздел (Пример N)
3. Code review
4. Merge в main

**Критерии качества примера**:
- ✅ Демонстрирует правильное применение S2R
- ✅ Содержит реальный контекст и объяснение
- ✅ Показывает структуру, отношения, метаданные
- ✅ Может быть использован как шаблон

### Приоритет примеров

**HIGH** — критичные для новичков:
- Создание нового ядра
- Создание документа в ядре
- Правильное именование
- Семейства документов

**MEDIUM** — для опытных пользователей:
- Связи между документами
- Миграция структуры
- FPF метаданные

**LOW** — специфические сценарии:
- Редкие типы документов
- Сложные зависимости
- Продвинутые техники
