# Интеграция S2R и FPF

> **Для ИИ-агентов:** Этот документ объясняет, как S2R использует FPF. Применяйте эти паттерны при работе с репозиторием.

---

## Принцип интеграции

S2R использует FPF как **эпистемологический фундамент**. Это означает:

1. **Терминология** — ключевые термины S2R берутся из FPF
2. **Паттерны мышления** — структура рассуждений следует FPF
3. **Онтология** — иерархия понятий опирается на FPF

---

## Ключевые паттерны FPF в S2R

### A.1 — Holonic Foundation (Холонический фундамент)

**Принцип:** Всё — холон: одновременно целое и часть большего целого.

**Применение в S2R:**
- Каждое ядро — холон (целое для своих семейств, часть репозитория)
- Каждый документ — холон (целое для своего содержимого, часть семейства)
- Репозиторий — холон (целое для проекта, часть экосистемы знаний)

**Примеры:**
```
Репозиторий ecosystem-development
├── Ядро A.Созидатель (холон)
│   ├── Семейство A2.1.Meaning (холон)
│   │   └── Документ "Ценность для созидателя" (холон)
```

**Почему важно:**
- Понимание уровней абстракции
- Корректное определение границ
- Управление сложностью через иерархию

---

### A.1.1 — Bounded Context (Ограниченный контекст)

**Принцип:** Значение термина локально — определено границами контекста.

**Применение в S2R:**
- Каждое ядро — отдельный контекст
- Локальная онтология ядра в `X0.{Name}-Management/local-ontology.md`
- При переносе терминов между ядрами — явное указание мостов

**Примеры:**

| Термин | Контекст A (Созидатель) | Контекст B (Экосистема) |
|--------|-------------------------|-------------------------|
| "Ученик" | Человек, проходящий путь созидателя | Участник экосистемы |
| "Платформа" | Место для создания проектов | ИТ-инфраструктура |
| "Компетенция" | Способность созидателя | Требование к наставнику |

**Почему важно:**
- Избегание семантических конфликтов
- Явные границы применимости
- Трассируемость терминов

**Как реализовано:**
- Frontmatter `kernel: A` указывает контекст
- Ссылки между ядрами через мосты в 0.2
- Локальные глоссарии в каждом ядре

---

### A.3 — System Triad (Триада систем)

**Принцип:** Для каждой целевой системы определяется надсистема и создатель.

**Применение в S2R:**
- Структура ядра: X1.Suprasystem, X2.SOI, X3.Constructor
- Обязательность определения всех трёх систем
- Матрица 3×3 как развёртка триады по ролям

**Примеры:**

**Ядро A (Созидатель)**:
- A1.Мир — надсистема (∋ Созидателя)
- A2.Созидатель — целевая система (SoI)
- A3.Экосистема — система создания (→ Созидателя)

**Ядро B (Экосистема развития)**:
- B1.Образовательный-рынок — надсистема (∋ Экосистемы)
- B2.Экосистема-развития — целевая система (SoI)
- B3.Система-создания-Экосистемы — система создания (→ Экосистемы)

**Почему важно:**
- Полнота системного описания
- Понимание контекста (надсистема)
- Понимание происхождения (создатель)

**Связь с F0-F9:**
- Надсистема (1) → F1, F2, F3
- SoI (2) → F4, F5, F6
- Создание (3) → F7, F8, F9

Подробнее: [0.3/roles-matrix.md](../0.3.Roles-Matrix-3x3/roles-matrix.md)

### A.7 — Strict Distinction (Строгое различение)

**Принцип:** Не путаем сущности разных типов.

**Применение в S2R:**

| Путаница | Правильно |
|----------|-----------|
| Документ "делает" | Система делает, документ описывает |
| Роль = Человек | Роль — функция, человек — исполнитель |
| План = Реальность | План — намерение, реальность — факт |
| Название = Сущность | Название указывает, сущность существует |

**Примеры ошибок:**

❌ **Неправильно:**
> "Документ 'Бизнес-модель' генерирует доход"

✅ **Правильно:**
> "Экосистема (система) генерирует доход. Документ 'Бизнес-модель' описывает, как это происходит"

❌ **Неправильно:**
> "Роль 'Визионер' — это Анатолий Левенчук"

✅ **Правильно:**
> "Анатолий Левенчук исполняет роль 'Визионер' в ядре A"

**Почему важно:**
- Избегание категориальных ошибок
- Ясность рассуждений
- Корректная трассируемость

**Как реализовано:**
- Префиксы документов (R.Role.md, C.Component.md, E.Event.md)
- Четкое различение в глоссарии
- Явные определения в онтологии

Подробнее: [0.1/07-naming.md](../0.1.Knowledge-Logic/07-naming.md)

### B.3 — Trust Calculus (Расчёт доверия)

**Принцип:** Доверие к информации оценивается кортежем F-G-R.

**Применение в S2R:**

| Компонент | Диапазон | Описание |
|-----------|----------|----------|
| **F** (Formality) | 0-9 | Формальность описания |
| **G** (Scope) | local/project/ecosystem | Область применимости |
| **R** (Reliability) | 0.0-1.0 | Надёжность, проверенность |

**Примеры:**

| Документ | F | G | R | Интерпретация |
|----------|---|---|---|---------------|
| ADR (Architecture Decision) | 7 | project | 0.9 | Формальное, проектное, проверенное решение |
| Черновик идеи | 2 | local | 0.3 | Неформальное, локальное, гипотеза |
| Спецификация API | 9 | ecosystem | 1.0 | Строгая, публичная, стабильная |

**Как реализовано в S2R:**
- **F** → `status` в frontmatter (draft/active/deprecated)
- **G** → `scope` в frontmatter (local/project/ecosystem)
- **R** → неявно через `status` + историю изменений

**Почему важно:**
- Оценка достоверности информации
- Принятие решений на основе надёжности
- Явное указание статуса документа

### E.10.D2 — I/D/S Discipline

**Принцип:** Различай три аспекта описания.

**Применение в S2R:**

| Аспект | Вопрос | Пример |
|--------|--------|--------|
| **I** (Intension) | Что это по сути? | "Авторизация — проверка прав" |
| **D** (Description) | Что наблюдаем? | "Используем JWT токены" |
| **S** (Specification) | Что требуется? | "Время ответа < 100ms" |

**Примеры в документах:**

**В архитектурном документе (F5):**
```markdown
## Авторизация (I)
Процесс проверки прав доступа пользователя к ресурсу.

## Текущая реализация (D)
- JWT токены с HS256
- Refresh tokens в Redis
- Session timeout: 1 hour

## Требования (S)
- Время ответа < 100ms (p95)
- Поддержка OAuth2
- Audit log всех попыток доступа
```

**Почему важно:**
- Разделение сущности, наблюдения и требований
- Избегание смешения уровней абстракции
- Ясность коммуникации

## Как использовать FPF при работе с S2R

### При создании документа

1. Определи, какие паттерны FPF применимы
2. Укажи их в `fpf_patterns` в frontmatter
3. Следуй принципам паттернов в содержимом

**Пример frontmatter:**
```yaml
---
type: spec
status: active
family: F5
fpf_patterns:
  - A.1       # Holonic Foundation
  - A.7       # Strict Distinction
  - E.10.D2   # I/D/S Discipline
---
```

### При анализе структуры

1. Проверь холоническую целостность (A.1)
2. Проверь границы контекстов (A.1.1)
3. Проверь строгость различений (A.7)

**Чек-лист:**
- [ ] Все ядра имеют 9 папок (3×3)? → A.1, A.3
- [ ] Локальные онтологии определены? → A.1.1
- [ ] Документы не путают сущности? → A.7
- [ ] Указан scope и status? → B.3

### При разрешении неясностей

1. Обратись к FPF-Spec.md за определением
2. Используй паттерны как критерии решения
3. Документируй решение со ссылкой на паттерн

**Пример:**
> "Согласно паттерну A.7 (Strict Distinction), различаем роль 'Наставник' (функция) и конкретного человека Ивана Петрова (исполнитель роли)."

## Ограничения контекста

FPF-Spec.md (~200K токенов) не вмещается целиком в контекст LLM. Стратегия:

1. **INDEX.md** — для обычных задач (~250 строк)
2. **Конкретные части** — для глубокого анализа (запрос по строкам)
3. **Ссылки на паттерны** — для документирования решений

**Рекомендация для ИИ:**

## Примеры применения паттернов

### Пример 1: Создание нового ядра

**Задача:** Создать ядро D для новой системы "Маркетплейс проектов"

**Применяемые паттерны:**
- **A.1** (Holonic Foundation) → Ядро D — холон репозитория
- **A.1.1** (Bounded Context) → Определить локальную онтологию D
- **A.3** (System Triad) → Определить D1 (надсистема), D2 (SoI), D3 (создания)

**Результат:**
```
D.Маркетплейс-проектов/
├── D1.Экономика-проектов/    # Надсистема
│   ├── D1.1.Meaning/
│   ├── D1.2.Architecture/
│   └── D1.3.Operations/
├── D2.Маркетплейс-проектов/  # SoI
│   ├── D2.1.Meaning/
│   ├── D2.2.Architecture/
│   └── D2.3.Operations/
└── D3.Команда-маркетплейса/  # Создания
    ├── D3.1.Meaning/
    ├── D3.2.Architecture/
    └── D3.3.Operations/
```

### Пример 2: Анализ документа

**Задача:** Проверить корректность документа "Архитектура платформы"

**Применяемые паттерны:**
- **A.7** (Strict Distinction) → Проверить, не путает ли документ систему и её описание
- **E.10.D2** (I/D/S) → Проверить разделение Intension, Description, Specification
- **B.3** (Trust Calculus) → Оценить F-G-R

**Чек-лист:**
- [ ] Документ различает "платформа" (система) и "схема платформы" (описание)? → A.7
- [ ] Есть секции I (что такое), D (как реализовано), S (требования)? → E.10.D2
- [ ] Указан `status`, `scope` в frontmatter? → B.3
- [ ] Указано `kernel: B` для контекста? → A.1.1
