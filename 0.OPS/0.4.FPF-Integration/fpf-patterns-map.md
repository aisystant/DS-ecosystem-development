# Карта паттернов FPF в S2R

## Детальная карта паттернов

### A.1 — Holonic Foundation

**Применяется к:**
- Репозиторий ecosystem-development
- Ядра A, B, C
- Семейства F0-F9
- Документы внутри семейств

**Примеры использования:**
```
Репозиторий (холон)
├── 0.OPS/ (холон F0)
│   └── 0.1.Knowledge-Logic/ (холон)
│       └── 01-kernels-model.md (холон)
├── A.Созидатель/ (холон)
│   └── A2.Созидатель/ (холон)
│       └── A2.1.Meaning/ (холон)
│           └── Ценность-для-созидателя.md (холон)
```

**Когда использовать:**
- При создании новой структуры
- При определении границ
- При анализе иерархии

**Связанные паттерны:** A.1.1 (Bounded Context)

---

### A.1.1 — Bounded Context

**Применяется к:**
- Локальные онтологии ядер
- Терминология внутри ядра
- Мосты между ядрами (0.2)

**Примеры использования:**

**Контекст A (Созидатель):**
```yaml
kernel: A
terms:
  - Ученик: человек, проходящий путь созидателя
  - Компетенция: способность созидателя
```

**Контекст B (Экосистема):**
```yaml
kernel: B
terms:
  - Ученик: участник экосистемы
  - Компетенция: требование к наставнику
```

**Когда использовать:**
- При определении терминов
- При переносе терминов между ядрами
- При анализе семантических конфликтов

**Связанные паттерны:** A.1 (Holonic Foundation), A.7 (Strict Distinction)

---

### A.3 — System Triad

**Применяется к:**
- Структура ядра (X1, X2, X3)
- Матрица 3×3
- Связи между системами (∋, →)

**Примеры использования:**

**Ядро A:**
```
A1. Мир (∋ Созидателя)
A2. Созидатель (SoI)
A3. Экосистема (→ Созидателя)
```

**Ядро B:**
```
B1. Образовательный-рынок (∋ Экосистемы)
B2. Экосистема-развития (SoI)
B3. Система-создания-Экосистемы (→ Экосистемы)
```

**Когда использовать:**
- При создании нового ядра
- При определении систем
- При анализе системных отношений

**Связанные паттерны:** A.1 (Holonic Foundation), 0.2 (Kernels Bridge)

---

### A.7 — Strict Distinction

**Применяется к:**
- Различение документ/система
- Различение роль/человек
- Префиксы документов (R., C., E.)

**Примеры использования:**

| Неправильно | Правильно |
|-------------|-----------|
| "Документ генерирует доход" | "Система генерирует доход, документ описывает" |
| "Роль Визионер = Анатолий" | "Анатолий исполняет роль Визионер" |
| "Plan.md выполняется" | "Система выполняется по Plan.md" |

**Когда использовать:**
- При написании документов
- При анализе формулировок
- При определении терминов

**Связанные паттерны:** A.1.1 (Bounded Context), E.10.D2 (I/D/S)

---

### B.3 — Trust Calculus

**Применяется к:**
- Frontmatter: status, scope
- Оценка достоверности
- Принятие решений

**Примеры использования:**

**Документ с высоким доверием:**
```yaml
status: active       # R = 0.9
scope: ecosystem     # G = ecosystem
type: spec           # F = 7-9
```

**Черновик:**
```yaml
status: draft        # R = 0.3
scope: local         # G = local
type: doc            # F = 2-4
```

**Когда использовать:**
- При создании документа
- При оценке надёжности информации
- При принятии архитектурных решений

**Связанные паттерны:** F.0.1 (Context Passport)

---

### E.10.D2 — I/D/S Discipline

**Применяется к:**
- Структура документов
- Архитектурные описания
- Спецификации

**Примеры использования:**

**В архитектурном документе:**
```markdown
## Авторизация

### Intension (I)
Процесс проверки прав доступа.

### Description (D)
Текущая реализация:
- JWT токены
- Redis для сессий

### Specification (S)
Требования:
- Время ответа < 100ms
- Audit log всех попыток
```

**Когда использовать:**
- При создании архитектурных документов
- При описании компонентов
- При написании спецификаций

**Связанные паттерны:** A.7 (Strict Distinction)

---

### F.0.1 — Context Passport

**Применяется к:**
- Frontmatter документов
- Метаданные
- Трассируемость

**Примеры использования:**

```yaml
---
type: spec                    # Тип документа
status: active                # Статус (R в B.3)
created: 2025-01-15           # Дата создания
updated: 2025-01-15           # Дата обновления
family: F5                    # Семейство (A.1)
kernel: B                     # Контекст (A.1.1)
scope: project                # Область (G в B.3)
fpf_patterns:                 # Используемые паттерны
  - A.1
  - A.7
---
```

**Когда использовать:**
- При создании любого документа
- При анализе метаданных
- При поиске документов

**Связанные паттерны:** B.3 (Trust Calculus), A.1.1 (Bounded Context)

## Паттерны по задачам

### Создание нового ядра

**Релевантные паттерны:**
1. **A.1** — определить ядро как холон
2. **A.1.1** — определить локальную онтологию
3. **A.3** — определить триаду систем (X1, X2, X3)

**Чек-лист:**
- [ ] Ядро имеет 9 папок (3×3)?
- [ ] Локальная онтология определена?
- [ ] Три системы идентифицированы?
- [ ] Связи ∋, → указаны?

### Создание документа

**Релевантные паттерны:**
1. **F.0.1** — заполнить frontmatter
2. **A.1.1** — указать kernel (контекст)
3. **A.7** — использовать строгие различения
4. **E.10.D2** — структурировать как I/D/S

**Чек-лист:**
- [ ] Frontmatter заполнен (type, status, family)?
- [ ] Kernel указан?
- [ ] Не путаем документ и систему?
- [ ] Есть секции I, D, S (если применимо)?

### Анализ структуры

**Релевантные паттерны:**
1. **A.1** — проверить холоническую целостность
2. **A.1.1** — проверить границы контекстов
3. **A.3** — проверить триаду систем
4. **A.7** — проверить строгость различений

**Чек-лист:**
- [ ] Все ядра имеют 9 папок?
- [ ] Локальные онтологии не конфликтуют?
- [ ] Три системы определены для каждого ядра?
- [ ] Нет категориальных ошибок?

### Разрешение конфликтов

**Релевантные паттерны:**
1. **A.1.1** — проверить контексты терминов
2. **A.7** — проверить строгость различений
3. **B.3** — оценить надёжность источников

**Чек-лист:**
- [ ] Конфликт терминов из разных контекстов?
- [ ] Конфликт из-за путаницы типов?
- [ ] Оценена надёжность (F-G-R)?

## Как указывать паттерны

### В frontmatter документа

```yaml
fpf_patterns:
  - A.1       # Holonic Foundation
  - A.7       # Strict Distinction
  - B.3       # Trust Calculus
```

### В тексте документа

```markdown
Согласно паттерну A.7 (Strict Distinction), различаем...
```

### В коммитах

```
feat(B2.1): добавлен Vision document (A.1, A.7)

Применены паттерны:
- A.1: документ как холон семейства F4
- A.7: строгое различение роль/человек
```
