# 0.4. FPF Integration

> **Для ИИ-агентов:** Этот раздел содержит интеграцию с First Principles Framework (FPF) — уровнем первых принципов, мета-эпистемой экосистемы. FPF задаёт правила корректного описания знаний, на основе которых строятся все предметные области (Episteme Packs).

Интеграция с **First Principles Framework (FPF)** — уровнем первых принципов экосистемы.

---

## Что такое FPF?

**First Principles Framework (FPF)** — это **мета-эпистема**, задающая правила корректного описания любых предметных областей. FPF — не методология, не курс и не предметная область, а **язык различений**, на котором строятся все остальные знания.

### FPF в архитектуре знаний

FPF занимает **уровень первых принципов** в четырёхуровневой архитектуре:

```
Уровень 1: FPF (Первые принципы)
     │
     ▼
Уровень 2: SPF (Фреймворк производства Pack'ов)
     │
     ▼
Уровень 3: Pack (Доменный source-of-truth)
     │
     ▼
Уровень 4: Downstream ← DS-ops здесь
```

### Fallback Chain для этого репозитория

**Принцип:** Если чего-то нет на текущем уровне — смотри выше.

```
1. Pack'и (source-of-truth) → PACK-ecosystem, PACK-digital-platform
2. SPF (форма и процесс)   → ~/Github/SPF/
3. FPF (первые принципы)   → ~/Github/FPF/ (только если нет в SPF)
```

**Когда нужен FPF напрямую:** редко. Только если SPF не покрывает вопрос о базовых различениях.

### Три ортогональных измерения

Помимо уровней, знания организуются по трём измерениям:

| Измерение | Что определяет | Роль FPF |
|-----------|----------------|----------|
| **Содержание** | Что описывать, терминология | FPF задаёт мета-терминологию |
| **Форма** | Как структурировать | S2R — один из форматов (не связан с FPF) |
| **Процесс** | Как производить знания | Определяется в SPF на основе FPF |

**Подробнее:** [[0.OPS/0.1.Knowledge-Logic/10-knowledge-architecture.md]]

**Источник FPF:** [github.com/ailev/FPF](https://github.com/ailev/FPF)

### Что FPF определяет

- **Как различать** практику, метод, инструмент, рабочий продукт
- **Как не путать** систему и процесс, область и её описание
- **Какие паттерны** использовать для корректного описания
- **Какие ошибки** избегать при построении знаний

### Что FPF НЕ является

| FPF — это | FPF — это НЕ |
|-----------|--------------|
| Мета-эпистема | Предметная область |
| Язык различений | Курс или руководство |
| Правила построения | Готовые знания |
| Уровень 1 архитектуры | Downstream-артефакт |

### Созидатель и FPF

Созидатель в экосистеме развивается **на основе FPF**. Это означает:
- Компетенции созидателя описываются с использованием FPF-терминологии
- Путь развития строится на FPF-паттернах мышления
- Все Episteme Packs (предметные области) созидателя опираются на FPF

## Расположение FPF

**Локальный путь:** `~/Github/FPF/FPF-Spec.md` (общий для всех репо)

FPF хранится централизованно для всех репозиториев экосистемы. Это обеспечивает:
- Единую версию FPF
- Простое обновление (`git pull`)
- Отсутствие дублирования

### Как обновить FPF

```bash
cd ~/Github/FPF && git pull
```

## Связь S2R и FPF

S2R **строится на основе FPF**, заимствуя:

| Концепция S2R | FPF-паттерн | Описание |
|---------------|-------------|----------|
| Ядро как холон | A.1 | Каждое ядро — целое и часть одновременно |
| Локальная онтология | A.1.1 | Bounded Context — термины локальны |
| Три системы | A.3 | System Triad: Suprasystem, SoI, Constructor |
| Строгое различение | A.7 | Роль ≠ Человек, Документ ≠ Система |
| Доверие | B.3 | F-G-R кортеж оценки информации |
| I/D/S дисциплина | E.10.D2 | Intension, Description, Specification |
